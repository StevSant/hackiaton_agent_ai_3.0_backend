# `data/` — what lives where

Single source of truth for everything the backend ships outside the codebase
itself: the canonical dataset, importable samples, demo documents, trained
models, and reference config. All files here are deterministic-by-design so
running the generator twice produces identical bytes.

---

## `synthetic/` — canonical generated dataset (the deliverable)

This is the dataset the jury sees and the one we hand to anyone who asks
"where did the data come from?". Built by:

```bash
uv run python scripts/generate_dataset.py
```

Outputs (all committed):

- `claims.json` — full `ClaimDetail` list with baked rule activations,
  score, tier, alertas, timeline. Loaded at app startup by the
  `SyntheticClaimQueries` adapter / DB loader.
- `siniestros.csv` — `siniestros` table per the §2.8 spec.
- `polizas.csv`, `asegurados.csv`, `beneficiarios_proveedores.csv`,
  `documentos.csv` — supporting tables.
- `demo_claims.json` — hand-authored demo cases (currently the SIN-2026-08412
  pack used in pitch screenshots). Generated by
  `scripts/import_frontend_demo_claims.py` when refreshed.

Names are real Ecuadorian (asegurados + providers) — pulled from the JSON
seed pools in `data/config/` (see below) via `_pools/` under
`app/use_cases/generate_dataset/`. No placeholders like `PROV-OBS-XXX` or
`Asegurado 3FE2` are emitted.

---

## `samples/` — size-variant importable files

For the analyst-importer UI (`POST /api/v1/claims/import`). All built from the
same `ARCHETYPES` cycle as `synthetic/claims.json` but at different sizes so
demos and tests can pick the right shape. Built by:

```bash
uv run python scripts/generate_samples.py
```

Variants (deterministic, non-overlapping ID ranges so they coexist in DB):

- `claims.dev.json`   — ~40 claims (fast local testing).
- `claims.test.json`  — ~80 claims (CI / demo rehearsal).
- `claims.prod.json`  — ~180 claims (near-realistic Supabase load).
- `claims.sample.csv` — first 10 rows of `dev` showing the CSV import format.
- `claims.demo-nuevo.csv` / `claims.sin-2026-08412.csv` — hand-crafted single-row
  examples used in the pitch demo flow.

CSV columns mirror `ClaimDetail`'s wire shape (Spanish `snake_case`).

---

## `sample_documents/` — demo PDF pack

PDF artifacts that ship with the SIN-2026-08412 demo claim (denuncia, acta
policial, matrícula, peritaje, etc). Used to render the "Documentos" tab in
the analyst UI and to demo the upload flow. Manifest at
`sample_documents/manifest.json`.

Regenerated by `scripts/generate_sample_claim_pdfs.py` if the templates
change.

## `config/` — reference / lookup data, hand-edited

JSON reference data the runtime *or* the synthetic generator needs but
that doesn't change per request. Edit the file, not Python code, when
a label or pool changes.

**Runtime config** (consumed by API endpoints):

- `ramo_labels.json` — `{canonical_ramo: display_label}` consumed by the
  insights donut + aggregation cards. Loaded once on startup via
  `app.domain.ramos.load_ramo_labels`.

**Generator seed pools** (consumed at dataset-build time only by
`scripts/generate_dataset.py` + `scripts/generate_samples.py`, via the
`app.use_cases.generate_dataset._pools` package). Each file wraps its
payload in `{"_comment": "...", "items": [...]}` (or `mapping` for dicts).
Order matters for list pools — the generator picks entries by index from
a deterministic hash.

- `nombres_masculinos.json` / `nombres_femeninos.json` — Ecuadorian first-name pools.
- `apellidos.json` — Ecuadorian surname pool.
- `proveedor_prefijos.json` / `proveedor_qualifiers.json` — building blocks for
  procedural provider names (e.g. "Taller Mecánico Cumbayá").
- `proveedores_default.json` — curated taller / repair-center names used as the
  default proveedor for vehicle claims when the archetype doesn't pin one.
- `code_provider_patterns.json` — prefixes that mark a 'proveedor' value as an
  internal code (PROV-OBS-…), triggering a substitution with a real name.
- `marcas_vehiculo.json` / `modelos_vehiculo.json` — vehicle brand + model pools
  (top sellers per SRI 2023-2024 stats).
- `sucursales.json` — `{ciudad: sucursal}` mapping for every city used in archetypes.
- `ramos_vehiculo.json` — set of ramo labels treated as vehicle claims (drives
  whether the generator emits a ClaimVehicle and picks a default proveedor).

When you add a new reference table, put it here and add a one-line entry
above so the next person knows what's in scope.

## `models/` — trained model artifacts

Binary outputs of the offline training notebooks (`notebooks/02_*.ipynb`).
Loaded at app startup by the LightGBM + Isolation Forest adapters.

- `fraud_lgbm.txt` — LightGBM Booster in native text format (inspectable).
- `anomaly_iforest.joblib` — Isolation Forest pickle.
- `anomaly_knn.joblib` — KNN nearest-neighbour index for the "nearest
  normal claim" UI affordance.

These are loaded only from this directory (root CLAUDE.md §17 — never from
user-controlled paths).

## Reproducibility

Re-generating everything from scratch:

```bash
# 1. Generate canonical synthetic dataset + per-variant samples
uv run python scripts/generate_dataset.py
uv run python scripts/generate_samples.py

# 2. (Optional) regenerate demo PDFs from templates
uv run python scripts/generate_sample_claim_pdfs.py

# 3. (Optional) retrain models — needs the notebooks env
uv run jupyter nbconvert --to notebook --execute notebooks/02_modelo_fraude.ipynb
```

`generate_dataset.py` prints a coverage report at the end — all 21 FS/RF
codes must have ≥3 exemplars or the generator fails the run.

## Anti-patterns

- ❌ Hand-editing `synthetic/*.csv` or `samples/*.json` directly — those are
  build outputs. Edit `_archetypes.py` (or the relevant generator input)
  and re-run.
- ❌ Committing real PII anywhere under `data/`. Synthetic / public only
  (root CLAUDE.md §2.10).
- ❌ Adding "fallback" data files that the runtime falls back to when the
  real source is unavailable — see the project-level rule: services either
  return real data or empty arrays, never hardcoded mocks.
